Splnit.eu Codebase Study Guide: How the App Is Built

Updated 2026-05-30 07:07. This version expands the earlier knowledge-base document with code-level explanations, implementation rationale, and guided reading notes.

How to use this document: read it next to the source code. Each section explains what a file does, why the pattern exists, what alternatives were avoided, and what to study next. It intentionally avoids secrets and does not reproduce every source file.
Repository state note: before this document edit, the repo already had unrelated dirty changes in package.json, package-lock.json, and untracked memory/. This document update does not resolve those.

1. Mental Model of the App

Splnit.eu is built as a multi-tenant compliance automation app. The user sees a product experience, but the code is organized around a few repeatable domain flows: tenant identity, intake/profile, framework/control mapping, evidence collection, document generation, vendor questionnaires, and trust-center publication.

Browser request → proxy.ts applies locale and Clerk auth boundaries → app/ route group renders public, app, auth, demo, API, or token route → server component or route handler calls lib/db/queries/* and domain logic in lib/* → domain module reads schema/data/config and returns typed data → component renders UI with localized copy and safe status labels → scripts/tests/smokes verify the behavior outside the browser
Why this choice: This split keeps framework-specific Next.js code in app/, reusable UI in components/, and business rules in lib/. That makes the compliance logic testable from scripts and route handlers instead of being trapped inside React components.

3. Request Entry: Root Layout and Proxy

3.1 app/layout.tsx

The root layout is global shell code. It loads fonts, sets metadata, attaches the next-intl provider, registers the service worker, and renders the cookie consent widget. It also injects a tiny script before hydration to avoid theme/locale flashes.

Why this choice: The theme/locale bootstrap is inline because it must run before React hydrates. If this were a normal client component, users could briefly see the wrong theme or locale. The layout also keeps metadata centralized so every page inherits a safe baseline.

5. Database Design and Access

5.1 Lazy database client

The DB client is intentionally lazy. The app should build without production secrets, but database-backed routes should fail clearly if DATABASE_URL is missing at runtime.

   9| let db: Database | null = null;
  10| 
  11| function getDatabaseUrl() {
  12|   const databaseUrl = process.env.DATABASE_URL?.trim();
  13| 
  14|   if (!databaseUrl || databaseUrl === '""' || databaseUrl === "''") {
  15|     return null;
  16|   }
  17| 
  18|   try {
  19|     const { protocol } = new URL(databaseUrl);
  20|     return protocol === "postgres:" || protocol === "postgresql:"
  21|       ? databaseUrl
  22|       : null;
  23|   } catch {
  24|     return null;
  25|   }
  26| }
  27| 
  28| export function hasDatabaseUrl() {
  29|   return Boolean(getDatabaseUrl());
  30| }
  31| 
  32| export function getDb(): Database {
  33|   const databaseUrl = getDatabaseUrl();
  34| 
  35|   if (!databaseUrl) {
  36|     throw new Error("DATABASE_URL is required for database access.");
  37|   }
  38| 
  39|   if (!db) {
  40|     if (isLocalDatabaseUrl(databaseUrl)) {
  41|       const localPool = new Pool({ connectionString: databaseUrl, max: 5 });
  42|       db = drizzleNodePostgres(localPool, { schema }) as unknown as Database;
  43|     } else {
  44|       db = drizzleNeon(neon(databaseUrl), { schema });
  45|     }
  46|   }
  47| 
  48|   return db;
  49| }
Why this choice: This avoids a common Next.js deployment problem: importing a module at build time initializes a database SDK before env vars are available. The code waits until getDb() is called. It also chooses node-postgres for local DB URLs and Neon HTTP for non-local URLs.

16. Safe Development Rules

  • Do not fabricate customers, testimonials, certifications, uptime, legal review status, metrics, or supported regulation coverage.
  • Do not expose internal evidence filenames, individual control IDs, exact test timing, or implementation details publicly in Trust Center pages.
  • Keep default locale as cs-CZ; do not silently make English or Italian the default.
  • Keep service clients lazily initialized to protect next build.
  • Use lib/stripe/plans.ts for Stripe plan names/prices instead of hardcoding them in UI.
  • For production claims, separate source/local proof, public route proof, authenticated tenant proof, and mailbox receipt proof.
  • When adding a new route under protected app surfaces, update/prove auth boundaries in proxy.ts.
  • When adding an integration, add provider types/registry/adapter/tests/storage/UI and focused smoke coverage; a card alone is not implementation.
